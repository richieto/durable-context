Why This Exists

Docs drift when they live apart from the code and depend on people remembering to update them. Code-Anchored Docs keeps shipped-behavior documentation in the repository, eventually consistent with the product, refreshed once per release from the tag diff.

Release-anchored

Docs at a tag describe the behavior of that tag — not the tip of a branch or an unfinished plan.

Independent

Release tags are the only requirement. Planning artifacts, if present, are optional enrichment.

Install: npx reference-docs init --project-name "My App". Default folder: reference/. Alternative: docs/reference/ if you prefer docs branding.

The Tag-Diff Workflow

The diff contains everything; the refresh step decides what changed in released behavior worth documenting.

1

Cut release tags

Tag each accepted release, e.g. release/v4_0_0.

2

Diff the tags

Scope to release/v4_0_0..release/v5_0_0.

3

Update affected pages

Only pages whose shipped behavior changed.

4

Record the refresh

Append a row to reference/releases/index.md.

Boundary Rules

Not during feature work

Docs are refreshed on request at release time, not as a side effect of every branch.

Diff is what you inspect

Refactors, formatting, and dependency bumps appear in the diff; judgment filters what gets documented.

Planning is optional

release-doc-notes.md from Durable Context can sharpen the refresh but is never required.
