Code-Anchored Context

Keep the thinking, not just the code

Coding agents can build and change software fast. But the most valuable part of that work is usually not the code that ships — it is the reasoning around it: why a decision was made, what was tried, what was deferred. Today that reasoning keeps disappearing. This is the case for keeping it.

The Problem: Valuable Thinking Keeps Escaping

When a team works with coding agents, a lot of careful thinking happens in places nobody owns. A chat window closes and it is gone. A document lives in one tool on one person's machine. None of it travels with the product. That is a quiet but real business risk.

Planning vanishes when a session ends

Hours of planning produce a clear direction, then the session closes and that direction is lost. The next person, or the next agent, starts from scratch.

Documentation drifts from reality

When docs live apart from the code and depend on each person remembering to update them, they slowly stop matching what the system actually does.

Knowledge is locked to one tool

If the plan and the context live inside one AI tool, you cannot ask a different one to review it, and switching tools means losing the trail. That is lock-in to a single vendor or setup.

Work cannot be handed off

Reasoning trapped in a session can only be acted on by that session. A specialist cannot review it, and a colleague cannot pick it up and continue cleanly.

The Idea: Keep Truth Close To The Code

The fix is simple to state. Keep the important context in the same place as the code it describes, structured so both people and agents can find it — and separate what already shipped from what the team is still deciding.

What shipped

Stable, release-accurate reference. It describes a known, delivered version of the product — not an unfinished change.

What we are deciding now

Living working context: what is being planned, built, decided, tested, shipped, and deliberately left for later. It is allowed to evolve as the team works through ambiguity.

Keep truth as close to code as possible, and keep the surrounding context structured enough that both humans and agents can find it.

Why It Matters

When context lives in the repository instead of a chat or a single tool, the benefits compound — for the team today and for whoever inherits the system tomorrow.

No tool lock-in

The context is portable. You can change AI tools, models, or vendors without losing the trail of why the system is shaped the way it is.

Knowledge the company owns

Understanding lives with the product, not in one person's head or one closed session. New people and new agents start from accumulated understanding.

Work you can delegate

A documented plan can be reviewed by a specialist before any code is written, handed to a colleague, or supervised with any agent.

Documentation that stays accurate

Because context ships together with the change it describes, it cannot quietly drift away from what the product actually does.

An audit trail for free

Living in the repository means every decision is version-tracked and reviewable, with no extra process to maintain.

Continuity through change

Tools, teams, and priorities change. The reasoning behind the system survives all of them.

The Current Direction

This is an evolving practice, not a finished product. The goal is steady and practical: make the reasoning around software durable by default, so that humans and coding agents share the same accumulated understanding over time — and so changing tools never means starting over.
