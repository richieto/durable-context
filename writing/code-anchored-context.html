Why This Exists

context/ is working development truth. decisions/ is the durable record of why the system is the way it is. reference/ is released product truth. Code-Anchored Context keeps agent context, planning, and decisions visible without mixing in-progress thinking into released reference.

Use context/ while work is forming

Put plans, specs, ADR drafts, backlog, and release-doc notes here during normal development. It is disposable; archive it when done.

Promote accepted choices to decisions/

The append-only ADR log outlives the bench, so the rationale behind accepted decisions stays findable after an initiative is archived.

Use reference/ only for shipped reference

Reference stays release anchored. Working context can feed it later through release-doc-notes.md.

Two independent packages ship this. Install either or both: npx @code-anchored-context/planning init brings context/ and decisions/; npx @code-anchored-context/reference init brings reference/. Reference works whether or not planning is used.

The Three Kinds Of Truth

The core rule is simple: put knowledge where its lifetime matches the work. Do not duplicate active plans into area-local folders.

Working context

Disposable bench, one folder per piece of work in progress. Drifts by design; archived when the work ships.

context/initiatives/<slug>/

Durable decisions

Append-only, numbered ADR log with statuses. Accepted decisions are promoted out of an initiative and kept here.

decisions/<NNNN>-short-title.md

Released reference

Release-tag-anchored behavior, owned by the reference package and refreshed from the diff between tags.

reference/

Where Does It Go?

Use this when deciding where new knowledge belongs.

Is this work in progress right now?
Use an initiative on the working bench. One folder per piece of work.
context/initiatives/<slug>/
Is this an accepted architecture or design choice?
Promote it into the durable, append-only decision log so it outlives the bench.
decisions/<NNNN>-short-title.md
Does this describe behavior that shipped in a release?
It belongs in release-anchored reference, refreshed from the tag diff.
reference/
Is it just future reference impact noted while building?
Leave it in the initiative as a hint for the next reference refresh.
context/initiatives/<slug>/release-doc-notes.md

Lifecycle

Work moves from a messy bench to durable artifacts. The planning skills are invocation-only — they run when you ask for them by name.

1

Start an initiative

Copy the initiative template into context/initiatives/<slug>/.

2

Plan with context

Use plan-with-context to draft the durable plan in plan.md across the whole change surface.

3

Grill and distribute

Use grill-and-distribute to fan settled truth into the per-concern docs.

4

Promote decisions

Accepted ADRs move out of the initiative into the decisions/ log.

5

Archive the bench

Once the durable truth is extracted, the initiative can be archived freely.

Delivery Surface

Working context describes the whole delivery system, not only the product code. The structure follows delivery concerns, not specific technologies or vendors.

testing.md

Verification strategy, automated and manual coverage, test data, regression risk, and release gates.

delivery.md

CI/CD behavior, build gates, deployment flow, environment promotion, release toggles, and delivery automation.

infrastructure.md

Environment shape, IaC, resources, networking, identity, storage, secrets, and environment dependencies.

operations.md is optional. Use it only for actionable runtime, support, observability, rollback, or repair context.

Mermaid diagrams are encouraged for flows and dependencies because they remain readable Markdown for agents and render as visible diagrams for humans.

Initiative Files

plan.md is the working surface. The other files are where stable conclusions belong.

File Purpose Stability
README.md Entry point, status, scope, touched areas, and links. Stable summary
plan.md Live alignment, questions, options, and notes to promote. Messy by design
spec.md What the system should do. Stable behavior
interface.md How humans, clients, APIs, config, reports, or tools interact. Stable surface
architecture.md Internal shape, boundaries, flows, contracts, and data. Stable design
testing.md Verification strategy, coverage, test data, and release gates. Stable verification
delivery.md CI/CD, build behavior, deployment flow, promotion, and toggles. Stable delivery
infrastructure.md Environment shape, IaC, resources, identity, storage, and secrets. Stable environment
operations.md Actionable runtime support: observability, failure modes, rollback, and repair. Optional operations
release-doc-notes.md Reference impact for later release-doc refresh. Bridge to reference

Key rule: plan.md can be messy, but settled truth must move into the stable files.

The Decision Log

decisions/ is a top-level, append-only ADR log. The bench is disposable, so accepted decisions are promoted here where they stay findable. To see what is in force, read the entries marked Accepted.

Rule What it means Where
One file per decision Numbered in order; copy the template to start a new entry. decisions/<NNNN>-short-title.md
Append-only Do not rewrite history. When a decision changes, add a new entry. Status: Superseded by NNNN
Status-driven Each entry is Accepted, Superseded, or Deprecated. Status: Accepted
Promoted from work Accepted decisions move out of an initiative's local ADR drafts. context/initiatives/<slug>/decisions/
