Why This Exists

Documentation/ is released product truth. Development/ is working development truth. Code-Anchored Context keeps agent context, planning, and decisions visible without mixing in-progress thinking into released product documentation.

Use Development/ while work is forming

Put plans, specs, ADRs, backlog, implementation context, future phases, and release-doc notes here during normal development.

Use Documentation/ only for shipped product docs

Product documentation remains release anchored. Development notes can feed it later through release-doc-notes.md.

The Four Work Containers

The core rule is simple: put knowledge where its lifetime matches the work. Do not duplicate active plans into area-local folders.

Program

Cross-release parent effort with durable context, roadmap, phase history, and program-level ADRs.

Development/programs/<program>/

Planned initiative

Scoped future delivery slice that belongs to a program but is not active yet.

programs/<program>/planned-initiatives/<initiative>/

Release initiative

Active or historical delivery slice for a specific release.

releases/<version>/initiatives/<initiative>/

Backlog item

Isolated work cut from scope and kept for possible later pickup.

backlog/items/<origin>--<item>.md

Decision Guide

Use this when deciding where new context belongs.

Is this active in the current release?
Use a release initiative. It is the delivery surface for the current release.
Development/releases/<current>/initiatives/
Does this span releases or phases?
Use a program for durable context, roadmap, and program-level decisions.
Development/programs/
Is a future phase already clear?
Use a planned initiative so detail is preserved before the release becomes current.
programs/<program>/planned-initiatives/
Was isolated work cut from scope?
Use a development backlog item. Link it back to the originating initiative.
Development/backlog/items/

Lifecycle

The same work can move through several containers as it becomes more concrete. Promotion is explicit and traceable.

1

Plan the arc

Create or update a program when the effort spans releases.

2

Scope future work

Capture clear future phases as planned initiatives.

3

Set current release

Run the release transition checklist and promote matching planned initiatives.

4

Deliver the slice

Work in the release initiative and keep stable files current.

5

Carry forward

Move durable context to programs and isolated cut scope to backlog items.

Delivery Surface

Development context describes the whole delivery system, not only the product code. The structure follows delivery concerns, not specific technologies or vendors.

testing.md

Verification strategy, automated and manual coverage, test data, regression risk, and release gates.

delivery.md

CI/CD behavior, build gates, deployment flow, environment promotion, release toggles, and delivery automation.

infrastructure.md

Environment shape, IaC, resources, networking, identity, storage, secrets, and environment dependencies.

operations.md is optional. Use it only for actionable runtime, support, observability, rollback, or repair context.

Mermaid diagrams are encouraged for flows and dependencies because they remain readable Markdown for agents and render as visible diagrams for humans.

Initiative Files

plan.md is the working surface. The other files are where stable conclusions belong.

File Purpose Stability
README.md Entry point, status, scope, touched areas, and links. Stable summary
plan.md Live alignment, questions, options, and notes to promote. Messy by design
spec.md What the system should do. Stable behavior
interface.md How humans, clients, APIs, config, reports, or tools interact. Stable surface
architecture.md Internal shape, boundaries, flows, contracts, and data. Stable design
testing.md Verification strategy, coverage, test data, and release gates. Stable verification
delivery.md CI/CD, build behavior, deployment flow, promotion, and toggles. Stable delivery
infrastructure.md Environment shape, IaC, resources, identity, storage, and secrets. Stable environment
operations.md Actionable runtime support: observability, failure modes, rollback, and repair. Optional operations
release-doc-notes.md Product documentation impact for later release-doc refresh. Bridge to docs

Key rule: plan.md can be messy, but settled truth must move into the stable files.

Release Transition

Changing Development/current.md is a workflow. The agent should materialize planned work for the new current release.

Step What happens Source
1 Update current.md and create the release folder if needed. _templates/release-context/
2 Scan programs for planned initiatives targeting the new release. programs/*/planned-initiatives/*/README.md
3 Promote matching planned initiatives into the release. releases/<version>/initiatives/
4 Update links both ways and leave the planned initiative as history. Status: Promoted
